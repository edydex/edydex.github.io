Technical Round

This tab is for the Viktor round: Linux, hosts, networking, hands-on lab work, script troubleshooting, and hardware sanity checks.

Good pattern: symptom -> scope -> command or check -> evidence -> fix -> verification.
Linux / Host Access
  • ssh user@host, ssh -i ~/.ssh/key.pem user@host, scp, rsync.
  • First confirm reachability, then auth, then service state on the box.
  • Best answer shape: “I check host health, then the service, then the logs, then the network path.”
Core Commands Worth Having Ready
  • uptime, top or htop, free -h, df -h, df -i
  • systemctl status <service>, journalctl -u <service>
  • ip addr, ip route, ss -tulpn, ping, curl
  • vmstat 1 5, iostat -xz 1 5 when CPU is not the whole story
Likely Question Strong Answer Shape Best Tie-In From You
“A Linux box is slow. What do you do?” Check load, CPU, memory, disk, inodes, then service logs and network blocking. Do not assume CPU is the problem. You already manage Linux systems and build service health checks.
“SSH is not working. Walk me through it.” Reachability, route, daemon status, logs, disk or inode exhaustion, auth versus network separation. Your general admin experience plus comfort checking logs before restarting.
“How would you debug a Bash or Python script failure?” Reproduce, inspect exact error or traceback, check exit code, environment, paths, permissions, dependencies, and service context. `mail_converter` setup/startup/validation/update scripts and Python service work.
“Explain VLANs / external networks.” Segmentation, access versus trunk, inter-VLAN routing, gateway and route validation, then DNS and service checks. You created a VLAN for IoT devices and security cameras.
“What if new lab hardware is not behaving?” Power, cable path, link lights, serial or management access, host visibility, logs, known-good baseline, one variable at a time. Use your current infrastructure ownership plus home lab instincts.

Networking Talking Points

  • VLAN = logical segmentation on shared switching infrastructure.
  • Access port carries one VLAN. Trunk carries multiple VLANs with tagging.
  • If devices are on different VLANs, confirm gateway and routing before blaming the host.
  • For packet issues: interface counters, route, DNS, listening port, then capture if needed.

Script Debugging Talking Points

  • Bash: bash -x script.sh, check $?, isolate syntax vs env vs permission vs dependency.
  • Python: read traceback carefully, confirm interpreter and packages, verify service user context.
  • Good line: “I want failures to be observable, not silent.”

RF / Home Lab

  • If RF comes up, be honest and connect it to physical-layer discipline and signal-path thinking.
  • Your Ubuntu and Debian home lab is a strong “Learn and Be Curious” and Linux credibility story.
Hardware / Power
  • Know server basics: CPU, RAM, NICs, PSUs, storage, fans, management ports.
  • Power formula: Watts = Volts x Amps.
  • 120V x 15A = 1800W theoretical; safe continuous target is often about 80 percent.
  • Redundant PSUs help only if the power paths are actually independent.
Bring-Up Answer

If I am bringing up new pre-production hardware, I start with power, connectivity, and a documented baseline. I verify cable paths, link state, management access, OS visibility, and logs. I change one variable at a time and document what good looks like so the process becomes repeatable for the rest of the team.

Strong technical-round tone: calm, methodical, low-ego, specific. If you say a command, explain why that command comes next.

`mail_converter` Project Story

This tab is the “talk like the actual builder/operator” view. It pulls from `mail_converter/docs/interview_prep.md`, which is much more specific than a resume bullet.

Best framing: AI helped with implementation, but you owned the architecture, debugging, integration, deployment, and product decisions.
60-Second Answer

`mail_converter` started as a PST and email-to-PDF conversion tool for e-discovery style workflows, but it grew into a larger Python system. It now has a conversion engine, a FastAPI web app, review and intake workflows, mailbox utilities like compare, merge, dedupe, and filter, a Linux deployment stack with `systemd`, PostgreSQL-backed authorization, Cloudflare Access in front, and backup and restore automation. The short version is that I built a Python-based email and document processing system with a real security model and a real Linux operations story.

Architecture In One Line

run_web.py -> webapp/server.py -> core/* + webapp/auth.py

  • Core: conversion pipeline, PST extraction, attachment conversion, mailbox output.
  • Web layer: FastAPI routes for conversion, intake, review, preview, downloads, health.
  • Auth layer: Cloudflare Access for identity, PostgreSQL-backed local authorization state.
  • Ops layer: installer scripts, service user, `systemd`, loopback-only DB, backups, restore.

Strongest Linux Example

  • Runs as a `systemd` service under a dedicated Linux user.
  • Loopback-bound app with Cloudflare tunnel and Access in front.
  • Installer creates `/opt`, `/var/lib`, `/var/log`, and env files under `/etc`.
  • Health checks prove readiness instead of assuming startup worked.

Strongest Python Example

  • Python as orchestration language around PST extraction, conversion, and web workflows.
  • FastAPI backend plus SQLAlchemy-backed authz model.
  • `subprocess.run` around external tools where that is the practical choice.
  • Targeted pytest coverage around auth/access, web safety, and file-conversion routing.

Strongest Operational Example

  • Scripts detect dependencies, create a venv, smoke-test imports, install the service, and poll /api/health.
  • Backup and restore are part of the deployment story, not an afterthought.
  • Good line: “I tried to make the system survivable outside my laptop.”
Likely Follow-Up Strong Answer
“What parts did you personally do if AI wrote a lot of code?” I owned the engineering loop: product decisions, architecture, security model, debugging, integration, testing, and deciding what had to change when the first pass was wrong.
“Why Cloudflare Access and PostgreSQL together?” They solve different problems: Cloudflare authenticates the user at the edge, while PostgreSQL stores app-specific authorization state like users, groups, invites, case permissions, and audit events.
“How does it boot on Linux?” Developer path: detect prerequisites, create a venv, install requirements, launch. Production path: service user, install under `/opt/mail_converter`, env files, `systemd` unit, start service, verify health.
“How do you know deployment worked?” Multiple layers: package install succeeds, critical imports succeed, `systemd` starts cleanly, and `/api/health` responds within timeout.
“What is still weak?” Dependency manifest drift, some operator-heavy Cloudflare setup, and long-term key-management and recovery hygiene. The architecture is real, but cleanup and hardening are still active work.
Honest AI-Assisted Framing

I built this as a solo AI-directed project. I used AI heavily for implementation, but I owned the product decisions, system design, debugging, feature direction, integration work, and the feedback loop that made the code actually fit the problem.

Why This Project Matters For Kuiper
  • Shows Linux service ownership, not just app coding.
  • Shows you think about access control, recovery, observability, and operator workflows.
  • Shows you can take a messy workflow and turn it into something repeatable and usable.
  • Shows you can go from concept to implementation to support.

Best project-closing line: “What I am proudest of is that it is not just a script that works on one machine. It has a real security model, a real Linux service story, and a real operational model.”

Leadership Principles

This round is about how you think and act. Use the official LP language, but anchor it in your own stories so it does not sound memorized.

Best LP story shape: short context, clear constraint, what you did, what happened, what you learned.
Likely LP Official Idea Your Best Angle
Ownership Leaders act on behalf of the entire company and never say “that’s not my job.” Sole IT administrator, vendor coordination, infrastructure continuity, and owning problems end to end.
Dive Deep Leaders operate at all levels, stay connected to details, and audit frequently. Linux troubleshooting, service logs, debugging scripts, and understanding `mail_converter` beyond the surface.
Bias for Action Speed matters. Many decisions are reversible and we value calculated risk taking. Infrastructure changes, fiber transition, rapid troubleshooting, and iterative improvement in your project work.
Earn Trust Leaders listen attentively, speak candidly, and treat others respectfully. Working with users, vendors, and stakeholders while documenting clearly and staying calm under pressure.
Invent and Simplify Leaders expect innovation and always find ways to simplify. Bash or Python tooling, health checks, repeatable setup, and productizing messy workflows.
Insist on the Highest Standards Leaders raise the bar and make sure problems are fixed so they stay fixed. Documentation, post-change verification, safer service configs, testing, and reducing regressions.
Deliver Results Leaders focus on the right inputs and deliver quality on time despite setbacks. Keeping infrastructure stable while shipping improvements and handling urgent issues.
Learn and Be Curious Leaders are never done learning and explore new possibilities. Your home lab, certifications, and expanding from admin work into deeper Linux, networking, and development ownership.
Behavioral Stories To Prepare
  1. A time you took a calculated risk: use the fiber transition or a meaningful infrastructure change.
  2. A time you solved an ambiguous problem: use Linux deployment, service hardening, or a messy project requirement.
  3. A time you improved a process: use scripts, health validation, update automation, or documentation.
  4. A time you worked across functions: use vendors, users, leadership, or project stakeholders.
STAR Reminders
  • Situation: give just enough context to make the stakes clear.
  • Task: explain what you personally were responsible for.
  • Action: spend the most time here.
  • Result: include outcome, verification, and what changed afterward.
  • End with what you learned if the result was mixed.

Good LP Language

  • “I wanted to leave behind a repeatable process, not just a one-time fix.”
  • “I narrowed the fault domain before making changes.”
  • “I documented the known-good state so the team could reuse it.”

What Hurts In LP Rounds

  • Talking vaguely without a real example.
  • Sounding like the team did everything and you did nothing specific.
  • Skipping the result or lesson.

Good Closing Habit

  • Finish with what changed after your action: uptime, clarity, speed, reduced risk, or easier support.
  • That makes the story feel operational and grounded.

LP shortcut: if you get stuck, choose one of your three strongest anchors: current sole-admin role, VLAN/network work, or `mail_converter` Linux service ownership.
