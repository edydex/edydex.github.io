Linux / Ubuntu Cheat Sheet

Keep your answers structured: establish symptoms, check host health, inspect service state, read logs, then verify the network path.

Task Commands What To Say
Access a host ssh user@host
ssh -i ~/.ssh/key.pem user@host
scp file user@host:/tmp/
rsync -avz dir/ user@host:/opt/app/		 “First I confirm reachability and credentials, then I use SSH to inspect the host directly. For file transfer I prefer scp or rsync depending on size and repeatability.”

Good Ubuntu Talking Points

  • System boot: BIOS or UEFI, GRUB, kernel, initramfs, systemd.
  • Service management and long-running processes: systemd.
  • Filesystem defaults you can mention: ext4 for stability, XFS for large-file throughput.
  • If memory pressure comes up: Linux can use page cache heavily; high usage is not always a leak.

Strong “SSH Not Working” Answer

  • Check reachability: ping, route, security group or firewall assumptions.
  • Check daemon: systemctl status sshd.
  • Check logs: journalctl -u sshd.
  • Check disk and inodes: full disks can break logins and services.
  • Verify auth issue versus network issue versus host issue.

Networking + VLANs

They do not need you to be a network architect. They want someone who can reason clearly through connectivity, segmentation, and lab reliability.

Topic Key Point Good Interview Language
VLAN basics Logical segmentation on the same physical switching infrastructure. “I use VLANs to separate traffic types, reduce blast radius, and apply clearer policy boundaries.”
Access vs trunk Access ports carry one VLAN. Trunks carry multiple VLANs with 802.1Q tagging. “Endpoints usually sit on access ports, while uplinks between network devices are commonly trunks.”
Inter-VLAN routing Traffic between VLANs needs a router or Layer 3 switch interface. “If two devices are in different VLANs, I confirm gateway and routing before assuming the host is broken.”
External network trouble Check local interface, gateway, DNS, routing, firewall, then the remote service. “I narrow the fault domain one hop at a time instead of guessing.”
Useful commands ip addr, ip route, ping, traceroute, dig, ss -tulpn, tcpdump -i eth0 “I verify addressing, routes, DNS, listening ports, and packet flow.”
Packet loss Could be interface errors, congestion, drops in the kernel, or the application itself. “I check interface counters and captures before blaming the network broadly.”
LEO / high-latency angle Expect higher latency and intermittent links; timeouts, retries, and buffering matter. “For higher-latency networks, I think about retry behavior, queueing, and avoiding assumptions that every timeout means total failure.”

Your real example: “I created a VLAN for IoT devices including security cameras. That gave me hands-on experience with segmentation, documentation, and making sure devices landed in the right place without disrupting the rest of the environment.”

Bash / Python Debugging

The brief says automation from scratch is not required. They mainly want to know that you can run scripts, inspect failures, and troubleshoot responsibly.

Scenario What To Run What To Say
Bash script failed bash -x script.sh
echo $?
shellcheck script.sh if available		 “I rerun with tracing, check the exit code, then isolate whether it is a syntax issue, permission problem, missing dependency, or bad environment variable.”
Python script failed python3 script.py
python3 -m pdb script.py
python3 -m pip list		 “I read the traceback carefully, confirm the interpreter and dependencies, and verify file paths, permissions, and runtime inputs.”
Script works manually but not as a service systemctl status app
journalctl -u app
sudo -u serviceuser env		 “I compare execution context: user, working directory, PATH, secrets, mounted paths, and network assumptions.”
Large log or file processing tail -f file.log
grep -n "ERROR" file.log
less, awk, or Python line streaming		 “I avoid loading large files into memory unless I need to. I prefer streaming and narrowing.”
External command from Python subprocess.run(..., check=True, capture_output=True, text=True) “I want explicit return codes and stderr so failures are observable instead of silent.”

Solid Debugging Flow

  • Reproduce the failure.
  • Read the exact error, exit code, or traceback.
  • Check inputs, permissions, paths, dependencies, and environment variables.
  • Check logs if the script is running under cron or systemd.
  • Make one change at a time and verify.

Hardware, Storage, and Power

Recruiter notes say hardware was heavily emphasized. Keep answers practical and safety-aware.

Server / Lab Hardware

  • Know the basics: CPU, RAM, NICs, PSUs, fans, storage, rails, and management ports.
  • Emphasize labeling, cable management, airflow, and documenting what is connected where.
  • Say you verify link lights, port mapping, serial or management access, and power redundancy.

Storage Devices

  • HDD: cheaper, slower, mechanical.
  • SATA SSD: faster than HDD, common general-purpose choice.
  • NVMe SSD: highest throughput and low latency over PCIe.
  • RAID: mention tradeoff between performance, redundancy, rebuild time, and usable capacity.

Cabling / Wiring

  • Separate power and data cleanly.
  • Label both ends.
  • Avoid strain, sharp bends, and blocked airflow.
  • For lab work, document cable maps and known-good configurations.
Power Topic Key Formula / Idea Good Interview Language
Volts, amps, watts Watts = Volts x Amps “I use volts times amps to estimate load and make sure we are not overcommitting power delivery.”
120V 15A example 120 x 15 = 1800W theoretical
Continuous safe target is often about 80 percent, roughly 1440W.		 “I care about the continuous load, not just the maximum printed rating.”
Redundant PSUs Spread A and B supplies across separate PDUs or circuits when possible. “Redundancy only helps if the power paths are actually independent.”
Before powering new equipment Check voltage requirements, connector type, amperage draw, breaker capacity, and upstream PDU. “I verify compatibility first because a power mistake can take down more than the single device.”
Troubleshooting a dead device Confirm outlet or PDU, PSU status LEDs, cable seating, switch state, and whether one PSU failed versus total power loss. “I isolate power source, cable, PSU, and device before assuming a board failure.”

If soldering comes up and your experience is limited: be honest, then pivot to adjacent hands-on strengths like cabling, rack work, hardware swaps, careful labeling, power checks, and following documented procedures around sensitive equipment.

What To Avoid

Answers That Hurt You

  • “I would reboot it first.”
  • Listing random commands with no reasoning.
  • Ignoring logs, service status, or disk and inode checks.
  • Claiming deep expertise you cannot defend.
  • Talking only about software when the question is clearly about hardware or lab operations.

Answers That Help You

  • Start with fault isolation.
  • Explain why each command matters.
  • Mention verification after the fix.
  • Talk about documentation, SOPs, alarms, and repeatability.
  • Use your real examples instead of generic textbook answers.

Best overall pattern: symptom -> scope -> command -> evidence -> fix -> verification -> prevention.
